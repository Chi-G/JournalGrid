# JournalGrid — Project AI Guidelines

> Append this file's content to the bottom of the `GEMINI.md` generated by
> `php artisan boost:install`. Boost's section covers idiomatic Laravel 13 /

> domain rules, which no generic guideline file can know.

## What this project is

JournalGrid is a double-entry journal voucher / general ledger entry system,
built to demonstrate `unnathianalytics/laragrid` (an Excel-style Livewire
datagrid) against the exact use case it was extracted from: a production
accounting system for spreadsheet-trained operators. Every architectural
decision here should read as "built by someone who understands both
accounting discipline and modern Laravel conventions" — not a CRUD tutorial.

## Non-negotiable domain rules

1. **Money is never a float.** All monetary columns are `bigInteger` minor
   units (kobo) or `brick/money` value objects at the boundary. Never
   introduce a `decimal` cast on a money column without discussing it first —
   floating-point rounding in a ledger is a correctness bug, not a style
   preference.
2. **A journal voucher's lines must always net to zero** (`Σdebit = Σcredit`)
   before the voucher can move out of `draft`. This is enforced in THREE
   places, not one: the LaraGrid `->completeWhenBalanced('debit', 'credit')`
   client guard, the `CreateJournalVoucherAction`/`PostJournalVoucherAction`
   server-side re-validation, and a DB constraint or scheduled reconciliation
   job. Never remove one of these layers because another already covers it —
   client-side checks are UX, not authority.
3. **Posted vouchers are immutable.** Never write code that updates a
   `journal_lines` row belonging to a voucher with `status = posted`. The only
   valid operation on a posted voucher is a reversal, which creates a NEW
   voucher (`reversal_of_id`) with mirrored debit/credit values. If asked to
   "edit a posted voucher," push back and suggest a reversal + new entry
   instead.
4. **Segregation of duties is a hard authorization rule, not a UI
   convenience.** The role that creates a voucher (Accountant) must not, by
   default, hold the permission to post it (Approver). Never write a policy
   or gate that collapses `voucher.create` and `voucher.post` into the same
   check "for simplicity."
5. **Voucher numbers are generated through `VoucherNumberGenerator`**, never
   the autoincrement `id`. Don't expose the primary key as a
   user-facing/business identifier anywhere (URLs, exports, printed
   vouchers).

## Architecture conventions (SOLID, applied)

- **Actions own business logic.** One invokable class per business
  operation: `CreateJournalVoucherAction`, `PostJournalVoucherAction`,
  `ReverseJournalVoucherAction`, `RecalculateTrialBalanceAction`. Livewire
  components and controllers call exactly one action and translate its result
  into UI state — they never contain `if` branches that encode accounting
  rules.
- **Events for side effects, not more conditionals in the action.**
  `VoucherPosted`, `VoucherReversed` dispatch listeners
  (`UpdateAccountBalances`, `NotifyApprover`, `WriteActivityLog`). Adding a new
  side effect means adding a listener, not editing `PostJournalVoucherAction`.
- **Constructor-inject collaborators.** Actions and services receive their
  dependencies (query objects, `VoucherNumberGenerator`,
  `ExporterRegistry`) via the constructor. Never `SomeService::query()`
  inline inside an Action — it can't be swapped or mocked in Pest tests if you
  do.
- **Policies are the single source of authorization truth.** Every
  `Grid::make(...)->authorize(...)` call, every controller, every queued job
  that touches a voucher references the same `JournalVoucherPolicy` methods.
  Never inline a `$user->hasRole(...)` check as a shortcut somewhere a policy
  method already exists.
- **DB transactions wrap every multi-row write.** A voucher header + its
  lines are one atomic unit — wrap creation, posting, and reversal in
  `DB::transaction()`.

## LaraGrid usage conventions for this repo

- Editable grids (voucher line entry) always declare `->rowsFrom()`,
  `->minRows(1)`, `->autoAppend()`, and `->completeWhenBalanced('debit',
  'credit')` together — this is the intended entry flow for this package and
  should not be replaced with a plain Livewire form + repeater.
- Readonly server-side grids (voucher list, GL by account) always declare
  `->authorize()` against the relevant Policy method, never a raw permission
  string duplicated ad hoc.
- The Trial Balance report is a **Display-mode** grid (rows passed to the tag,
  not `->query()`-backed) because it's a computed aggregate, not a queryable
  table — don't "simplify" it into a readonly server-side grid; that's the
  wrong mode for this data shape.
- Any new picker column (`SearchSelectColumn`) that enriches a row via
  `->onSelect()` must keep that enrichment logic in the Grid definition's
  closure, not smuggled into a model observer — the grid's docs are explicit
  that hooks are the authoritative half of the optimistic-update model, and
  scattering that logic elsewhere breaks the "one place to look" guarantee.

## Testing expectations

- Every Action gets a Pest unit test that does NOT boot Livewire — test the
  business rule (balance enforcement, immutability, segregation of duties) in
  isolation.
- Every grid gets a Pest feature test using `Livewire::test(...)->call(
  'gridOps', ...)` mirroring the pattern in the LaraGrid README — test through
  the public RPC surface, not by reaching into private component state.
- Never mark a posting/reversal test as passing without asserting the
  resulting `total_debit_minor`/`total_credit_minor` actually reconcile to
  zero net change across the GL.
